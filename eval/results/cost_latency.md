# Cost & latency

Generated by `eval/cost_model.py` (run it yourself to reproduce or tweak any
assumption) plus the retrieval latency actually measured in this repo's
`eval/run_eval.py` run. Managed-pricing figures are cited; re-verify before
using this for a real budget decision, since vector-DB pricing moves often.

## Monthly cost at 100K / 1M / 10M vectors

All rows model **384-dim vectors** (bge-small-en-v1.5, the recommended
production embedder for this project) plus ~1200 bytes/chunk of stored
metadata + chunk text, since that's what this project actually keeps
alongside each vector. "Lightly queried" = 1,000 queries/day throughout.

| Vectors | Self-hosted embedded Qdrant (this project) | Pinecone Serverless (2026, realistic bill) | Pinecone legacy dedicated pods | Qdrant Cloud (managed, same engine) |
|---|---|---|---|---|
| 100K  | **$18.03/mo** | $0/mo (fits free tier) | $70/mo (1 pod) | $0/mo (fits free tier) |
| 1M    | **$18.29/mo** | $50/mo (Standard plan floor) | $70/mo (1 pod) | ~$30/mo+ (Standard tier floor) |
| 10M   | **$20.94/mo** | $50/mo (Standard plan floor) | $140/mo (2 pods) | ~$98/mo (scaled estimate) |

Self-hosted cost is nearly **flat** across three orders of magnitude of
vector count: a small fixed VM (~$18/mo) dominates the bill, since disk for
even 10M vectors at this dimensionality is only ~33GB (~$3/mo). This only
holds because query volume is genuinely light and the deployment relies on
on-disk (mmap) storage rather than requiring the whole index resident in
RAM - see trade-offs below.

## Sources for the managed-pricing figures

<cite index="2-1">Pinecone's Standard plan bills storage at roughly $0.33 per GB per month, with write units starting around $4 per million and read units starting around $16 per million</cite>. <cite index="4-1">Pinecone's free tier covers about 2GB of storage (roughly 350K vectors uncompressed), and dedicated legacy pods range from about $70/month for the smallest tier up to $700+/month for high-performance pods</cite>. <cite index="5-1">Independent cost-modeling work quotes the same Standard-tier numbers and notes a $50/month minimum applies to that paid plan regardless of usage</cite>. <cite index="6-1">Pinecone has fully committed to serverless pricing as of 2026, with pod-based indexes now considered legacy and no idle compute charge under the new model</cite> - notably, this is the exact problem this assignment's background describes, and the market has partly already moved to solve it for lightly-queried workloads.

For the same-engine managed comparison, <cite index="11-1">Qdrant Cloud's 2026 pricing includes a permanent free tier (0.5 vCPU, 1GB RAM, 4GB disk) plus a Standard tier billed hourly for dedicated compute and memory, typically landing in the $30-200/month range</cite>, and <cite index="12-1">third-party estimates put a production 3-node cluster with replication for around 5M vectors at 768 dimensions at roughly $65-130/month</cite>. Qdrant Cloud doesn't publish a flat per-vector rate card - actual cost flows through their resource calculator - so the 1M/10M rows above are scaled estimates, not quotes.

## Retrieval latency (measured, this repo)

From `eval/run_eval.py`, 22 queries against the real ingested corpus (37 chunks, embedded query -> Qdrant search, no LLM call), embedded local Qdrant, spaCy embedder:

| Stage | p50 | p95 | mean |
|---|---|---|---|
| Embed query | 6.6 ms | 7.7 ms | 7.0 ms |
| Retrieve (Qdrant search) | 0.51 ms | 0.59 ms | 0.52 ms |
| **Total (embed + retrieve)** | **7.2 ms** | **8.3 ms** | **7.5 ms** |

This is genuinely fast because everything is local and in-process - no
network hop to a managed service, no cold start. It will not stay this fast
forever: this was measured against a 37-chunk corpus; Qdrant's HNSW search
itself scales sub-linearly with corpus size, but embedding time is roughly
constant, so total per-query latency should stay in a similar ballpark even
at 100K-10M vectors as long as the index stays resident (RAM or hot page
cache). *Generation latency is not included here* - this evaluation run
used a stub LLM client (no `GROQ_API_KEY` was available in the build
sandbox, and `api.groq.com` isn't reachable from it either - see README), so
end-to-end latency including a real Groq call is not yet measured; re-run
`eval/run_eval.py` with a real key to get that number, since it will
typically dominate total request latency (a real LLM call is usually
hundreds of milliseconds to a couple seconds on Groq's fast inference,
versus single-digit milliseconds for retrieval).

## Trade-offs accepted by self-hosting (as asked for explicitly)

- **No built-in HA / replication / automatic backups.** A managed service
  bundles this; a single self-hosted node is a single point of failure
  unless you build replication yourself (a real, non-trivial engineering
  cost not reflected in the dollar figures above).
- **Embedded mode specifically is single-process.** `QdrantClient(path=...)`
  takes an exclusive file lock - fine for one service instance, not for
  multiple writers/readers across processes. Scaling that out means running
  Qdrant as its own server process (still self-hosted, still roughly the
  same infra cost story, but a different deployment shape than this repo's
  embedded default).
- **You own upgrades, monitoring, and capacity planning.** No vendor SLA,
  no automatic scaling if query volume spikes unexpectedly.
- **The flat self-hosted cost assumes genuinely light query volume.** If
  queries-per-day grew by 100x, the small fixed VM would become the
  bottleneck (CPU-bound HNSW search, or RAM-bound if the working set no
  longer fits page cache), and the "flat cost" story would break down -
  at that point the self-hosted side would need to scale compute, not just
  disk, and the comparison should be re-run with that assumption changed.

## When this project's Discussion answers "when would you switch back to managed"

See the top-level README's Discussion section - the short version is: once
query volume (not vector count) grows enough that you're paying engineering
time to keep the self-hosted node reliable, or once you need multi-region/HA
guarantees this single-node embedded setup doesn't provide.
